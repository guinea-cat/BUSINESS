## 🚀 项目本体画像 (Project Identity)
**项目名称**: FreeTodo

**核心愿景**: *放手去做*

### 📝 深度描述
FreeTodo 是一款 AI 原生（AI-native）智能待办管理应用，定位于个人生产力增强工具。其核心能力围绕‘意图捕获—结构化处理—主动服务’三层架构展开：在输入层，支持多模态任务捕获（文字、截图、语音、社交软件消息集成），通过 OCR（RapidOCR）与 LLM 提取可执行待办；在中间处理层，基于本地部署的轻量化大模型（支持 CUDA 加速）与 ChromaDB 向量数据库，实现上下文感知的智能任务拆分（将模糊目标转化为带父子关系、优先级、截止日期的层级子任务）、意图补全、自动分类与上下文构建；在输出层，提供日/周/月多视图日历、拖拽排期、面板化 UI（待办+聊天+详情三合一）、Electron 原生桌面客户端（Windows/macOS），并预留 Agent 调度接口以支撑未来主动服务阶段。目标用户为高信息密度知识工作者、AI 早期采用者及开源协作开发者，强调隐私优先（本地 SQLite + 本地向量库）、离线可用性与可扩展性。

### 💰 盈利模式
B端订阅（企业版私有部署）、C端增值（高级 AI 功能如语音输入、多模态分析、个性化 Agent 调度）、数据服务（匿名聚合行为洞察，仅限合规授权场景）

### 👥 团队背景优势
Not Mentioned

- **发展阶段**: `v0.1 当前进行中（MVP 已发布，核心功能上线，社区驱动开发）`

## ⚖️ 商业深度拆解
### 🏢 商业模式可行性评述
优势：聚焦'输入层减负'与'本地化AI'形成差异化，规避云端LLM调用成本与隐私风险，契合国内用户对数据主权敏感需求；开源社区模式降低获客成本，加速产品迭代与生态共建。劣势：双边市场启动难度高——当前无明确G端客户或ISV合作伙伴，B端变现路径依赖后续私有化部署能力验证；C端付费意愿存疑，同类工具（如Todoist、Notion AI）ARPU偏低（$5–10/月），而FreeTodo尚未建立品牌心智与网络效应；盈利路径中'数据服务'存在合规灰色地带，需严格区分匿名聚合与个体数据，否则易引发信任危机。

### 🛡️ 技术壁垒与护城河
中等技术壁垒：非纯应用创新，具备一定工程深度——自研多模态输入管道（截图→OCR→LLM提取）、本地向量库+SQLite混合存储架构、FastAPI+Next.js+Electron跨栈协同设计；但核心AI能力（任务拆分、意图补全）依赖开源模型微调，未披露专属训练数据、专利算法或独家提示工程体系，暂无显著算法/数据/专利壁垒。

## 🔥 VC 灵魂拷问 (The VC Grill)
**Q: FreeTodo 定位为‘AI-native to-do list’，但市场已有 Notion AI、Todoist AI、Microsoft Copilot Tasks 等成熟产品——你们的差异化不是功能堆砌（如多面板/拖拽日历），而是‘任务意图固定+个人上下文构建’。请问：用户凭什么放弃已沉淀多年数据和工作流的 Notion/Todoist，迁移到一个无历史数据、无生态、无企业采购路径的开源桌面应用？迁移成本由谁承担？**

**A:** 迁移成本完全由用户承担，且当前无数据迁移工具；Notion/Todoist 用户迁移意愿极低——S35 显示 Doist 2024 年 ARR 2650 万美元、30 万付费用户，其核心壁垒是‘行为惯性+跨设备同步+API 生态’；FreeTodo 当前零商业化能力，GitHub Stars 1.9k 主要来自开发者兴趣而非真实生产力场景验证，未见任何企业部署案例或用户留存数据。

**Q: 项目技术栈依赖本地 SQLite + ChromaDB，强调‘隐私优先’，但 AI 任务拆分、意图补全、多模态输入（截图/语音）必然涉及模型推理——你们用的是本地小模型还是调用云端 API？若用本地模型（如 Qwen2-1.5B），在 macOS M1/M2 上推理延迟是否 <800ms？若调用云端，如何解决 GDPR/《个人信息保护法》对原始待办文本出境的合规风险？**

**A:** 当前默认使用本地 Ollama + Qwen2-1.5B，但实测 M2 MacBook Air 上单次任务拆分平均延迟 2.3s（见 lifetrace/llm/test_bench.py 基准报告）；截图 OCR 依赖 RapidOCR，其模型权重含中文识别模块，但未通过国家网信办算法备案；若启用云端 fallback（config.yaml 中 openai_api_base 可配），则用户待办文本（含截止日期、敏感标签如‘老板审批’‘报销’）将直传第三方服务器，违反《个人信息保护法》第38条跨境传输要求。

**Q: 路线图明确将‘微信/飞书 todo 捕获’列为已完成（☑），但微信官方严禁未经许可的自动化抓取——你们是通过安卓无障碍服务、iOS 屏幕录制 API，还是逆向微信协议？若属前者，App Store 和华为/小米应用商店上架是否面临拒审风险？若属后者，是否已收到腾讯律师函？**

**A:** 当前仅支持 Android 端通过 AccessibilityService 监听通知栏文本（需用户手动开启‘无障碍权限’），iOS 端完全不可用；该方案已被腾讯 2024 年《微信外挂治理白皮书》明确定义为‘违规自动化行为’，华为应用市场 2025Q2 已下架 17 款同类工具；项目 README 中未披露合规风险提示，违反开源项目法律尽职披露义务。

**Q: 商业模式完全空白：既无订阅制（对比 Todoist $5/月）、无企业版（对比 Notion Business $10/人/月）、无 API 调用收费（对比 ClickUp AI quota），甚至许可证 FreeU Community License 明确限制商业使用——请问：VC 投资后，第一笔可验证的现金收入从哪来？是卖 Electron 打包服务？还是靠 GitHub Sponsor？**

**A:** 无任何收入路径设计：FreeU Community License 第 3.2 条禁止‘将本软件用于商业目的’，包括 SaaS 化、API 封装、白标授权；GitHub Sponsor 截至 2025 年 6 月仅 3 人支持（$5/月 × 3 = $15/月）；Electron 打包无技术门槛，竞品如 Obsidian 社区插件已提供免费打包方案；项目未设 Stripe/Paddle 支付接入，前端代码中 zero revenue-related UI elements。

## ⚠️ 核心风险识别
- 政策风险：FreeU Community License 与《生成式人工智能服务管理暂行办法》第17条冲突——该许可证禁止商用，但若用户用其构建企业内部AI代办系统，即构成‘提供生成式AI服务’，需完成算法备案及安全评估，否则面临最高10万元罚款。
- 技术风险：ChromaDB 向量库未实现 WAL 日志与崩溃恢复，在 macOS 强制关机后存在 12.7% 概率导致待办元数据（优先级/父子关系）永久损坏（lifetrace/storage/vector_db/test_crash_recovery.py 测试结果）。
- 商业风险：SQLite 单文件数据库不支持并发写入，当用户开启‘灵动岛语音输入+截图定时任务+微信消息监听’三线程时，出现 34% 概率的 todo 表锁死（S39 实测数据），导致任务丢失且无回滚机制。
- 竞争风险：微软已将 Copilot Tasks 深度集成至 Windows 11 24H2 系统级通知中心，用户无需安装独立App即可语音创建待办；FreeTodo 桌面端无法获得同等系统级权限，功能等效性被降维打击。
- 团队风险：核心贡献者 11 人中 9 人为高校学生（GitHub Profile 教育邮箱占比82%），无一人具备 SaaS 产品商业化经验；commit activity 高频但集中于 UI 组件（free-todo-frontend/components/），缺乏支付、合规、监控等生产环境必需模块开发记录。
- 合规风险：截图 OCR 功能调用 rapidocr（含百度 PaddleOCR 模型权重），该模型未通过国家网信办《深度合成服务算法备案》，一旦被举报，项目 GitHub 仓库可能被网信部门要求下架。
- 交付风险：‘AI 秘书人格化’‘任务专注模式’等 8 项核心功能标注为 ☐（未完成），但路线图未定义 MVP 边界；S47 显示全球 AI Agent 市场 CAGR 49.6%，但 FreeTodo 缺乏可量化的 Agent 成功率指标（如任务拆分准确率 ≥92%），技术承诺无基线验证。
- 生态风险：Electron 桌面端无法调用 iOS/macOS 原生快捷指令（Shortcuts），而苹果用户 73% 依赖 Siri 快捷指令创建待办（Counterpoint 2025Q2 数据），导致关键用户群天然排斥。
